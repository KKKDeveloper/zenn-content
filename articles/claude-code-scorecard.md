# Claude Codeを入れた。で、設定は？

Claude Codeをインストールして、ターミナルで `claude` と叩く。動く。コードも書いてくれる。便利だ。

でも、1ヶ月後にこう思っていないだろうか。

> 「思ったほど生産性上がってないな」

原因はたいてい、Claude Codeの性能ではない。**設定の問題**だ。

Claude Codeは、CLAUDE.md・rules・hooks・skills・memoryの5層を正しく設計すると、同じモデルでも出力品質が劇的に変わる。逆に言えば、設定なしで使っているチームは、**性能の30〜50%しか引き出せていない**。

この記事では、自分のClaude Code環境を10カテゴリ・100点満点で採点できるスコアカードを公開する。

## 前提：この記事で扱う範囲

[前回の記事](https://zenn.dev/aim_kimura/articles/5f643cd172b2f3)で、組織でClaude Codeを運用するための方法論「Claude Code Flow」を紹介した。

今回はその実践編。**自分の設定がどのレベルにあるか**を定量的に把握するためのチェックリストだ。

個人利用でもチーム利用でも使える。まずは自分のスコアを出して、どこが弱いかを知ることから始めてほしい。

# スコアカード（全10カテゴリ・100点満点）

## 1. CLAUDE.md設計（10点）

CLAUDE.mdはClaude Codeの「初期プロンプト」にあたる。プロジェクトを開くたびに自動で読み込まれる、最も重要な設定ファイルだ。

| # | チェック項目 | 配点 |
|---|---|---|
| 1-1 | プロジェクトルートにCLAUDE.mdがある | 2 |
| 1-2 | サブディレクトリにもCLAUDE.mdを配置している | 2 |
| 1-3 | CLAUDE.mdの合計行数が200行以内に収まっている | 3 |
| 1-4 | 詳細はスキルファイル等に分離し、CLAUDE.mdは索引に徹している | 3 |

**なぜ200行か？**
CLAUDE.mdが長すぎると、Claudeは後半の指示を無視しやすくなる。自分たちの経験では200行を超えたあたりから守られない指示が出始めた。短く保ち、詳細は別ファイルに逃がすのが鉄則。

:::message
**0点の人へ**: まず `CLAUDE.md` を1つ作るだけで変わる。プロジェクトの概要、使用技術、コーディング規約の3つを書くだけでいい。
:::

## 2. ルール体系（10点）

`.claude/rules/` にルールファイルを置くと、CLAUDE.mdとは別に自動読み込みされる。CLAUDE.mdを短く保ちつつ、詳細なルールを定義できる。

| # | チェック項目 | 配点 |
|---|---|---|
| 2-1 | `.claude/rules/` にルールファイルが存在する | 2 |
| 2-2 | ルールがカテゴリごとにファイル分割されている | 3 |
| 2-3 | コーディング規約・命名規則がルールに含まれている | 2 |
| 2-4 | セキュリティに関するルールが定義されている | 3 |

**実践例**:
```
.claude/rules/
├── 01-identity.md      # プロジェクトの基本情報
├── 02-coding.md        # コーディング規約
├── 03-security.md      # セキュリティルール
├── 04-review.md        # コードレビュー基準
└── 05-testing.md       # テスト方針
```

番号付きでファイルを分割すると、何がどこにあるか一目でわかる。チームメンバーが「ルールを変えたい」と思ったとき、該当ファイルだけ編集すればいい。

## 3. Hooks（安全装置）（10点）

Hooksは「Claude Codeがツールを使う前後に自動実行されるスクリプト」だ。人間がうっかりOKを押しても、Hookが最後の砦になる。

| # | チェック項目 | 配点 |
|---|---|---|
| 3-1 | PreToolUse hookが1つ以上設定されている | 2 |
| 3-2 | `git push --force` や `rm -rf` を防止するHookがある | 3 |
| 3-3 | シークレット情報の誤コミットを検出するHookがある | 3 |
| 3-4 | Hookがファイル内容まで検査している（ファイル名だけでなく） | 2 |

**これが一番差がつくカテゴリ**。Hookを1つも設定していないチームは、Claude Codeに本番環境を触らせるのと同じだ。

具体的なHookの例：

```bash
#!/bin/bash
# PreToolUse hook: force pushを防止
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

if echo "$COMMAND" | grep -qE 'git\s+push.*(-f|--force)'; then
  echo "BLOCKED: force pushは禁止されています" >&2
  echo '{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny"}}'
  exit 0
fi
exit 0
```

このスクリプトを `settings.json` に登録するだけで、Claude Codeがforce pushを実行しようとした瞬間に自動ブロックされる。人間の確認を挟む余地すらなく、技術的に不可能になる。

:::message alert
**Hookなし = シートベルトなしで高速道路に乗るようなもの。** 事故が起きてからでは遅い。最低でも「force push防止」と「secrets検出」の2つは入れよう。
:::

## 4. スキル・コマンド（10点）

`.claude/commands/` にMarkdownファイルを置くと、`/コマンド名` で呼び出せるカスタムコマンドになる。チームの業務フローをコマンド化することで、「Claudeへの指示の仕方」を標準化できる。

| # | チェック項目 | 配点 |
|---|---|---|
| 4-1 | `.claude/commands/` にカスタムコマンドがある | 2 |
| 4-2 | 業務に特化したコマンドが5本以上ある | 3 |
| 4-3 | コマンドのdescriptionにトリガーキーワードが設定されている | 2 |
| 4-4 | 複数エージェントが連携するパイプラインがある | 3 |

**コマンド化の効果**:
- 「レビューしておいて」→ `/review` と打つだけ
- 新人でもベテランと同じ品質の指示が出せる
- 指示の品質が属人化しない

## 5. メモリ・ナレッジ（10点）

Claude Codeのメモリ機能を使うと、会話をまたいで情報を蓄積できる。プロジェクトの文脈、過去の判断、チームの方針などを覚えさせることで、毎回のコンテキスト説明が不要になる。

| # | チェック項目 | 配点 |
|---|---|---|
| 5-1 | memory/ にプロジェクト情報やフィードバックを蓄積している | 2 |
| 5-2 | メモリにカテゴリ分類（feedback/project/reference等）がある | 3 |
| 5-3 | メモリの棚卸し・整理ルールが定義されている | 3 |
| 5-4 | knowledge/ に業務ドキュメントを整備している | 2 |

メモリは便利だが、放置すると腐る。古い情報が残ったまま新しい会話に影響すると、むしろ害になる。**月1回の棚卸しルール**を決めておくことが重要。

## 6. セキュリティ（10点）

Claude Codeはファイルの読み書き、コマンド実行、Git操作ができる。つまり、**設定を間違えると機密情報を漏らす能力がある。**

| # | チェック項目 | 配点 |
|---|---|---|
| 6-1 | `.env` や秘密鍵が `.gitignore` に含まれている | 3 |
| 6-2 | `settings.local.json` が `.gitignore` に含まれている | 2 |
| 6-3 | シークレットは環境変数参照を徹底している | 3 |
| 6-4 | 機密情報を含むメモリファイルが個別に `.gitignore` 指定されている | 2 |

見落としがちなのが `settings.local.json`。Claude Codeを使っていると、ツール許可のたびにこのファイルにSSH先のIPアドレスやキーパスが記録されていく。`.gitignore` に入れ忘れると、リポジトリ経由でサーバー情報が漏洩する。

## 7. 失敗学習（Gotchas）（10点）

Gotchas = 「Claude Codeがよくやらかすパターン」の記録。これがあるかないかで、チームの学習速度が決定的に変わる。

| # | チェック項目 | 配点 |
|---|---|---|
| 7-1 | gotchas（失敗パターン）を記録するファイルがある | 3 |
| 7-2 | gotchasに「症状・対策・出典」の構造がある | 3 |
| 7-3 | gotchasを参照するタイミングがルールで定義されている | 2 |
| 7-4 | gotchasの卒業（ルール昇格）基準がある | 2 |

**実践例**:
```markdown
### APIキーのハードコード
- **症状**: コード内に直接APIキーを埋め込んで生成する
- **対策**: 環境変数 `os.environ["VAR_NAME"]` で参照させる
- **出典**: 2024-12の本番障害
- **ステータス**: ルール昇格済み（→ rules/03-security.md）
```

成功事例より、**失敗パターンの蓄積**が組織の生産性を決める。ルールに昇格した（＝自動で守られるようになった）ものは「卒業」扱いにして、gotchasファイルを肥大化させない運用も大事。

## 8. 自動化（10点）

Claude Codeは非対話モード（`claude -p`）でスクリプトから呼び出せる。定型作業を自動化すると、「AIに指示を出す」という行為すら不要になる。

| # | チェック項目 | 配点 |
|---|---|---|
| 8-1 | cronや定期実行でClaude Codeを非対話的に使っている | 3 |
| 8-2 | 自動化スクリプトにログ出力・ローテーションがある | 2 |
| 8-3 | MCP接続エラー時のリトライルールが定義されている | 3 |
| 8-4 | 作業ログを自動・半自動で記録している | 2 |

自動化の例:
- 毎朝5時にカレンダー・タスク・Slackを集約してデイリーレポート作成
- PRが作成されたらClaude Codeが自動レビュー
- 週次でSEO記事を自動生成・下書き保存

## 9. コンテキスト管理（10点）

Claude Codeのコンテキストウィンドウは有限。大きなファイルを読み込んだり、長い会話を続けると、初期の指示が押し出されて精度が落ちる。

| # | チェック項目 | 配点 |
|---|---|---|
| 9-1 | サブエージェントのモデル（haiku/sonnet/opus）を使い分けている | 3 |
| 9-2 | コンテキスト肥大化時の引き継ぎルールがある | 2 |
| 9-3 | 大量出力を検知する仕組み（hook等）がある | 3 |
| 9-4 | settings.jsonの許可パターンがワイルドカードで整理されている | 2 |

**モデルルーティングの目安**:
| モデル | 用途 | コスト感 |
|---|---|---|
| Haiku | ファイル検索、リスト取得、単純変換 | 安い・速い |
| Sonnet | コード修正、文書作成、ほぼ全タスク | バランス |
| Opus | 複雑な設計判断（原則使わない） | 高い・遅い |

Agent toolの `model` パラメータを明示するだけで、不要なコスト消費を防げる。

## 10. 独創性・組織適応（10点）

ここまでの9カテゴリは「基盤」。最後のカテゴリは、**自分たちの業務にどこまでフィットさせているか**を問う。

| # | チェック項目 | 配点 |
|---|---|---|
| 10-1 | 自社の業務フローに特化した独自スキルがある | 3 |
| 10-2 | チーム全員がClaude Codeを活用している（属人化していない） | 3 |
| 10-3 | 他ツール（Notion, Slack, Calendar等）とMCP連携している | 2 |
| 10-4 | 定期的にスキル・ルールを見直す運用サイクルがある | 2 |

# スコア判定

合計点を出したら、以下の表で自分の位置を確認してほしい。

| スコア | 判定 | 解説 |
|---|---|---|
| **90〜100** | S: 世界トップレベル | Claude Codeの可能性を最大限引き出している。事例として公開すべきレベル |
| **75〜89** | A: 先進的 | 主要機能を活用。あと一歩の仕組み化で大きく伸びる |
| **50〜74** | B: 発展途上 | 基本は押さえているが、Hooks・Gotchas・メモリが未整備 |
| **25〜49** | C: 初期段階 | CLAUDE.mdは書いたが組織的な活用に至っていない |
| **0〜24** | D: 未活用 | インストールしただけ。設定の力を知らない |

## 自社でやってみた結果

筆者（AIM LLC）の環境で採点した結果：**97点**。

内訳と、高得点の背景を参考までに公開する。

| カテゴリ | スコア | 背景 |
|---|---|---|
| CLAUDE.md設計 | 95 | root + 5サブディレクトリの6ファイル構成。合計188行 |
| ルール体系 | 98 | 8ファイルにモジュール分割。CxO自動介入という独自ルールあり |
| Hooks | 100 | 6つの安全装置。ファイル内容のシークレットパターン検出まで実装 |
| スキル・コマンド | 96 | 58コマンド・48スキル。ビジネス5段階+エンジニアリング5段階パイプライン |
| メモリ | 95 | 63ファイル。月次棚卸しルール・evolution昇格フローあり |
| セキュリティ | 97 | settings.local.json含む全機密ファイルを.gitignore。Hook二重チェック |
| Gotchas | 100 | 26エントリ。症状/対策/出典の統一構造。卒業ルールあり |
| 自動化 | 95 | cron朝ブリーフィング、VPSメモリ同期、23本のスクリプト |
| コンテキスト管理 | 98 | モデルルーティング、PostToolUse大量出力警告Hook |
| 独創性 | 95 | CxO自動介入、Gotchas学習ループ、8専門家ペルソナ |

1年以上の運用で、このスコアになった。最初から97点だったわけではない。最初は30点程度で、Gotchasを貯めながら改善を繰り返してここに至った。

## 効果（Before → After）

| 指標 | Before | After | 改善率 |
|---|---|---|---|
| 提案書作成 | 8〜12時間/件 | 2〜3時間/件 | 70%短縮 |
| 朝のタスク確認 | 30分（手動） | 3分（自動） | 90%短縮 |
| セキュリティ事故 | 四半期1回 | 0件（6ヶ月間） | 100%排除 |
| 同じミスの再発 | 月2〜3回 | 月0〜1回 | 80%削減 |

特にHooksによるセキュリティ事故ゼロは、数字以上のインパクトがある。「APIキーを漏らすかもしれない」という心理的不安がなくなるだけで、Claude Codeに任せられる作業の範囲が大幅に広がった。

# スコアを上げるための優先順位

全部いきなりやろうとすると挫折する。優先順位はこうだ。

### まず今日やること（0点→40点）
1. **CLAUDE.md を作る**（+4点）
2. **force push防止Hookを入れる**（+5点）
3. **`.gitignore` にsettings.local.jsonと.envを追加する**（+5点）

この3つで15分。それだけで40点台になる。

### 今週やること（40点→65点）
4. `.claude/rules/` にセキュリティルールを分離する
5. 業務でよく使う指示を `/commands/` にコマンド化する（3本）
6. gotchasファイルを作り、直近で起きた失敗を3つ記録する

### 今月やること（65点→80点）
7. メモリのカテゴリ分類を整備する
8. サブエージェントのモデルルーティングを設定する
9. MCP連携を1つ試す（Slack or Notion）
10. 月次の棚卸しルールを決める

### じっくり育てる（80点→90点超）
11. 業務特化スキルを10本以上作る
12. 自動化スクリプトを組む
13. チーム全員に展開する
14. Gotchasの卒業サイクルを回す

# まとめ

Claude Codeの設定は、一度作って終わりではない。使いながらGotchasを貯め、ルールに昇格させ、Hookで自動化する。この**学習ループ**が回り始めると、AIの精度が勝手に上がっていく。

スコアカードで自分の現在地を把握して、まずは「今日やること」の3つから始めてみてほしい。
